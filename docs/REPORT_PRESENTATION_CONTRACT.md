# Report Presentation Contract

## Статус и назначение
Этот контракт замораживает правила teacher-facing presentation для текущего berchun MVP. Он управляет только тем, как текущая математическая правда показывается в итоговом отчёте и на его графиках.

Reference-style target:
- accepted reference `references/DZ1.pdf` остаётся визуальным ориентиром;
- текущий `report/final_report.pdf` и связанные машинно-читаемые артефакты остаются источником математической истины.

## Граница действия
Контракт распространяется только на:
- `report/final_report.tex` / `report/final_report.pdf`;
- captions, notes, condition blocks, explanation blocks;
- visual skin существующих data-driven plots;
- локальную teacher-facing последовательность изложения.

Контракт не разрешает менять:
- solver mathematics;
- `out/data` truth;
- figure data / series content;
- scheme semantics;
- run archive, intake или build behavior.

## Политика отображения чисел
- Обычные teacher-facing числа в тексте, captions, scheme notes, variable blocks, plot notes и plot labels показываются максимум с `3` знаками после запятой.
- Лишние нули и завершающая точка удаляются.
- Целочисленные параметры и count-like величины (`Tc`, `Ts`, `Tw`, `N`, `n`, `m`, `r`) остаются точными целыми.
- Если ненулевое значение при округлении стало бы `0.000`, допускается форма `< 0.001` вместо длинной десятичной дроби.
- Научная запись допускается только в явных диагностических местах, где она сама является смысловой частью отчёта.
- `epsilon = 1e-12` в `1.4`;
- residual/tail bounds в `1.4`;
- иные явно диагностические величины того же класса.
- Внутренняя точность в JSON и solver outputs не меняется.

## Политика блока `Исходные данные`
- Каждый item начинается с визуально единообразного блока `Исходные данные:`.
- Заголовок блока всегда выделяется жирно.
- Значения внутри блока подаются как короткие отдельные строки без лишней prose-обвязки.
- Формат блока должен быть одинаковым во всех `1.1`, `1.2`, `1.3`, `1.4`, `2.1`.
- Числа в этом блоке подчиняются общей numeric policy.

## Политика читаемости графиков
- Ни одна legend не должна накладываться на plotted data, оси, заголовки осей или explanatory notes.
- Для family plots legend размещается вне plotting field либо в зарезервированной верхней зоне, а не поверх данных.
- Для single-series plots legend опускается, если caption и оси уже однозначно объясняют график.
- Внутренний plot title не должен дублировать caption тяжёлой синтетической фразой. Если caption уже идентифицирует график, внутренний title либо сокращается, либо убирается.
- Grid, spines и цветовая иерархия должны помогать чтению, а не перетягивать внимание с данных.
- Визуальный стиль должен оставаться академичным и спокойным, без декоративных эффектов.

## Политика показа нестационарной области
- Для `1.3` нестационарная область должна оставаться явно отделённой.
- Decision frozen: нестационарная область сохраняется как visual marker, но только в виде светлой нейтральной вертикальной подложки, не агрессивной по цвету и не спорящей с основной кривой.
- Для нестационарных точек не рисуются выдуманные stationary values.
- Первый стационарный `n` обязательно проговаривается рядом в prose.
- Объяснение значения shading даётся один раз локально в блоке `1.3`, а не размазывается как шум по нескольким графикам.

## Политика объяснения переменных
- Сразу после каждой расчётной схемы должен идти короткий локальный блок `Обозначения:` или эквивалентный по смыслу.
Локальный блок обязан объяснить:
- смысл состояний схемы;
- все параметры, которые появятся в ближайших формулах;
- все дополнительные локальные обозначения вроде `a`, `\rho_n`, `P_wait`, если они используются дальше.
- Читатель не должен восстанавливать значения символов по памяти из предыдущих разделов.
- Если символ вводится заново в другом item, его локальное объяснение повторяется.

## Политика плотности derivation flow
- Большие формульные пачки разбиваются на меньшие локальные блоки.
- Между соседними displayed formulas должна быть короткая человеческая связка, если иначе теряется смысл перехода.
- Не допускаются длинные участки из более чем `3` displayed formulas подряд без prose-перехода.
- После ключевого derivation block должен идти хотя бы один локальный числовой checkpoint, привязанный к текущим артефактам.
- Dense spots в `1.1`, `1.4` и `2.1` считаются первоочередными для teacher-facing reflow.
- Новая математика не вводится; меняется только порядок и плотность изложения.

## Политика captions
- Caption должна быть короткой, учебной и естественной по языку.
- Caption описывает, что именно изображено, а не пересказывает весь смысл раздела.
- Интерпретация остаётся в prose рядом с фигурой, а не внутри caption.
- Нумерация figures должна оставаться детерминированной и монотонной.
- Caption family не должна возвращаться к synthetic/repetitive wording вида «метрика по числу ...» там, где можно сказать короче и естественнее без потери смысла.

## Политика human readability
Каждый item должен читаться как последовательность:
- условие;
- `Исходные данные`;
- схема;
- `Обозначения`;
- derivation;
- локальный числовой checkpoint;
- figure;
- интерпретация.
- Плотные абзацы разбиваются так, чтобы текст был удобен для устной защиты.
- Числовые выводы формулируются коротко и локально, рядом с соответствующим графиком или derivation block.
- Ни одна ключевая переменная не должна появляться в prose или caption без локального объяснения в том же item.

## Неизменяемые математические инварианты
Следующие вещи frozen и не подлежат изменению в polish-passes:
- solver outputs и traceability chain;
- `out/data` truth;
- current validated threshold truth для `1.1`;
- explicit non-stationary handling в `1.3`;
- `epsilon = 1e-12` и current residual-bound truth в `1.4`;
- arrival-weighted meaning of `waiting_probability` в `2.1`;
- figure data content и scheme semantics;
- canonical raw-input driven autofill.

## Запрещённые действия в будущих polish-passes
- менять solver code ради более «красивых» teacher-facing чисел;
- вручную подменять результаты в prose/captions;
- рисовать данные или stationary values там, где текущие артефакты их не подтверждают;
- жертвовать математической правдой ради визуального сходства с reference;
- смешивать report-polish passes с feature work по build/archive/platform layers.
