# Methodical Guide Plan

## Статус и назначение
Этот документ замораживает `M0 — Methodical Guide Roadmap Freeze` для новой отдельной ветки проекта: methodical guide branch.

Назначение ветки:
- построить поверх уже стабильного report baseline отдельный student-facing guide;
- объяснять курсовую работу максимально простым русским языком;
- опираться на реальный вариант студента и реальные вычисленные значения;
- помогать не только читать работу, но и защищать её устно.

Этот guide не заменяет формальный отчёт и не меняет его статус как teacher-facing deliverable.

## Почему это отдельная ветка, а не переписывание отчёта
- Формальный отчёт уже выполняет другую функцию: он показывает решение в академическом и teacher-facing формате.
- Methodical guide решает иную задачу: проводит слабого студента по логике решения от реального смысла к формулам, графикам и выводам.
- Смешивание этих двух поверхностей в одном артефакте быстро ухудшит обе:
  - отчёт станет перегружен учебной и защитной prose;
  - guide потеряет право на более медленный, объясняющий и повторяющий стиль.
- Отдельная ветка позволяет менять объяснительную структуру без reopening solver truth, report contract и teacher-facing presentation contract.

## Целевой пользователь
- Студент с очень слабой подготовкой по теории массового обслуживания.
- Читатель, который путается в обозначениях и не держит длинную математическую цепочку в памяти.
- Пользователь, которому нужен не только итог, но и пошаговое понимание: «что происходит», «зачем эта формула», «что показывает график», «что сказать на защите».

## Учебная цель
Guide должен доводить студента до состояния, в котором он:
- понимает, какую реальную систему моделирует каждый подпункт;
- видит, как из исходных данных появляются интенсивности, состояния, вероятности и метрики;
- не теряется при переходе `схема -> обозначения -> формулы -> графики -> вывод`.

## Цель для защиты
Guide должен помогать студенту:
- коротко объяснить смысл каждой модели своими словами;
- привязать формулы к реальному процессу, а не заученной записи;
- читать собственные графики и называть их практический смысл;
- отвечать на базовые устные вопросы по своему варианту и своим числам.

## Границы ветки
Ветка methodical guide:
- работает поверх существующего solver/report/build baseline;
- использует уже вычисленные артефакты и зафиксированные contracts как source of truth;
- упрощает язык и порядок объяснения, но не меняет математическую правду.

Ветка methodical guide не должна:
- заменять `report/final_report.pdf`;
- вводить новую математику;
- подменять variant-specific числа общими примерами;
- становиться скрытым redesign текущего teacher-facing report branch.

## Почему guide должен быть variant-aware
- Устная защита проходит по собственному варианту студента, а не по абстрактному примеру.
- Слабому студенту особенно важно видеть свои числа, иначе объяснение остаётся чужим и быстро распадается на заученные фразы.
- Variant-aware guide позволяет прямо связывать:
  - исходные параметры варианта;
  - вычисленные метрики;
  - конкретные графики;
  - короткие защитные объяснения.
- Это убирает ручную послеправку вида «подставь тут своё число» и снижает риск расхождения между guide и formal report.

## Будущая модель генерации
На planning level future guide замораживается как generated artifact поверх успешного run bundle.

Будущая генерация должна опираться на:
- canonical raw-input identity студента;
- derived parameters варианта;
- machine-readable outputs из `out/data/*.json` или run-bundle эквивалентов;
- current figure set и figure manifest;
- formal report structure и contracts как ordering/reference layer.

Ожидаемая логика сборки:
1. Берётся конкретный run студента.
2. Для каждого подпункта `1.1`, `1.2`, `1.3`, `1.4`, `2.1` извлекаются реальные variant-specific значения и ключевые выводы.
3. Поверх этих артефактов собираются простые объясняющие блоки и defense notes.
4. Итоговый guide остаётся согласованным с formal report, но говорит более простым языком и в более учебном порядке.

Формат финального вывода здесь не замораживается глубже, чем нужно для roadmap: в M0 фиксируется именно content contract и generation direction, а не export surface.

## Frozen Roadmap

### M1 — Methodical Section Skeleton + Artifact Mapping
Результат:
- заморожен канонический skeleton будущего guide;
- для каждого подпункта зафиксировано, из каких именно артефактов берутся числа, графики и смысловые checkpoints;
- отделены blocks `объяснение`, `вариантные данные`, `защита`, `простыми словами`.

Почему это первым:
- без явной artifact map guide легко превратится в prose-only текст без проверяемой опоры на реальные значения;
- M1 делает следующую генерацию reviewer-safe.

### M2 — Variant-Aware Core Explanation Generation
Результат:
- реализуется минимальная generation path для простого объяснения по `1.1`-`1.4` и `2.1`;
- guide начинает собираться из реальных variant-specific данных, а не из hardcoded текста;
- на каждый подпункт появляются blocks про смысл модели, обозначения, формулы и графики.

### M3 — Defense Layer Integration
Результат:
- в каждый подпункт добавляются короткие defense notes;
- guide получает слой «что могут спросить / что коротко ответить / где обычно путаются»;
- блоки защиты привязываются к тем же variant-specific числам, а не живут как общий appendix.

### M4 — End-to-End Assembly + Consistency Validation
Результат:
- guide собирается end-to-end для реального run;
- проверяется согласованность чисел, обозначений, графических ссылок и локальных выводов с formal report и machine-readable data;
- определяется канонический operator path для future guide generation.

## Что считается успехом для этой ветки в целом
Methodical branch считается идущей в правильном направлении, если future guide:
- объясняет ту же задачу, что formal report;
- делает это заметно проще;
- остаётся variant-aware;
- помогает устной защите;
- не вносит расхождений с report/output truth.

## Следующий implementation target
После M0 следующий правильный шаг: `M1 — Methodical Section Skeleton + Artifact Mapping`.
