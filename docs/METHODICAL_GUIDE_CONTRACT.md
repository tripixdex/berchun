# Methodical Guide Contract

## Статус и назначение
Этот контракт замораживает правила для будущего generated methodical guide.

Guide понимается как:
- отдельный student-facing explanatory layer;
- артефакт поверх текущего formal report baseline;
- помощник для понимания и устной защиты;
- variant-aware документ, а не общий теоретический конспект.

Guide не является заменой formal report и не становится новым источником математической истины.

## Целевой читатель
- Почти без предварительного понимания ТМО и марковских моделей.
- Плохо удерживает длинные цепочки формул без словесного сопровождения.
- Может читать только короткими порциями и часто теряет смысл обозначений между абзацами.

## Тон и язык
Будущий guide обязан быть написан:
- максимально простым русским языком;
- короткими предложениями;
- без скрытых смысловых скачков;
- без жаргона без немедленного пояснения;
- без фразы «это очевидно» и аналогичных предположений о сильной подготовке читателя.

Обязательные речевые правила:
- сначала бытовой смысл, потом термин;
- сначала смысл символа, потом сам символ;
- одна новая идея на один короткий абзац;
- если термин неизбежен, его смысл объясняется в той же локальной точке;
- каждый подпункт заканчивается слоем `Что это значит простыми словами`.

## Порядок объяснения
Для каждого содержательного подпункта explanation order замораживается так:
1. Что происходит в реальной системе.
2. Что именно требуется найти в этом подпункте.
3. Какие исходные данные относятся к текущему варианту и что они означают.
4. Какие состояния, объекты и роли есть в модели.
5. Какие обозначения будут использоваться дальше.
6. Почему формула имеет именно такой вид.
7. Что означает вычисленная метрика.
8. Что показывает график и как его читать.
9. Что это значит простыми словами.
10. Что можно коротко сказать на защите.

Guide не должен перескакивать напрямую:
- от формулы к графику без словесного моста;
- от набора обозначений к выводу без промежуточного смысла;
- от общих слов к variant-specific числам без явного перехода.

## Обязательные разделы guide

### Guide-level sections
Будущий guide обязан содержать:
- краткое объяснение, что это за документ и чем он отличается от formal report;
- snapshot текущего варианта;
- последовательный walkthrough по `1.1`, `1.2`, `1.3`, `1.4`, `2.1`;
- встроенные defense notes;
- короткий финальный блок про то, как пользоваться guide на защите.

### Mandatory subsection blocks
Каждый подпункт `1.1`, `1.2`, `1.3`, `1.4`, `2.1` обязан содержать:
- `Что происходит в системе`;
- `Что дано в моём варианте`;
- `Что означают состояния и обозначения`;
- `Почему формула именно такая`;
- `Как читать график`;
- `Что это значит простыми словами`;
- `Что сказать на защите`.

## Правила объяснения формул
- Ни один символ не появляется без локального определения до первого использования.
- Перед формулой даётся короткая словесная подводка: что именно сейчас считаем и зачем.
- После формулы даётся короткое объяснение её смысла обычными словами.
- Если формула опирается на предыдущую, переход проговаривается явно.
- Ключевые formula blocks должны сопровождаться хотя бы одним variant-specific числовым checkpoint.
- Guide объясняет смысл формулы, но не притворяется новым математическим proof-layer сверх уже зафиксированного решения.

## Правила объяснения графиков
- Перед графиком guide обязан назвать, что находится по оси X и что измеряется по оси Y.
- Должно быть явно сказано, что меняется, а что остаётся фиксированным.
- После графика guide обязан объяснить форму кривой простыми словами.
- Между формулой и графиком всегда нужен словесный мост: почему именно из этой метрики получается такой график.
- В объяснении графика должны использоваться реальные точки или характерные значения текущего варианта, а не абстрактные placeholders.
- Guide не имеет права ссылаться на график как на «самоочевидный».

## Правила подачи defense notes
Defense layer является обязательной частью branch и не может быть отложен в отдельный необязательный appendix.

Для каждого подпункта defense notes должны быть представлены в коротком, повторяемом формате:
- какой вопрос могут задать;
- какой короткий ответ считается достаточным;
- где студент обычно путается;
- какие слова лучше не подменять неточным жаргоном.

Defense notes должны:
- опираться на тот же variant-specific материал, что и основное объяснение;
- помогать говорить по существу, а не заучивать чужой шаблон;
- оставаться короткими и пригодными для устного повторения.

## Допустимые источники данных и артефактов
Future guide может опираться только на проверяемые артефакты текущего baseline:
- canonical raw input и student identity fields;
- derived parameters варианта;
- machine-readable outputs для задач;
- текущие figures и figure manifest;
- formal report source/output для структуры и cross-check;
- contracts, которые уже замораживают formal report behavior.

Приоритет источников:
1. machine-readable outputs и variant-derived data для чисел;
2. current report contracts для структуры и терминологической совместимости;
3. current formal report surface для порядка изложения и cross-check.

Guide не должен вводить новые числа, которые не подтверждаются этими артефактами.

## Что обязано оставаться согласованным с formal report
- тот же вариант и те же входные данные;
- та же математическая правда;
- те же ключевые выводы по режимам;
- те же graph identities и те же линии рассуждения там, где они подтверждены артефактами;
- та же последовательность подпунктов coursework.

Guide может менять:
- тон;
- плотность объяснения;
- порядок локальных пояснений внутри подпункта;
- наличие blocks `простыми словами` и `на защите`.

Guide не может менять:
- solver truth;
- formal report contract;
- report figures/data semantics;
- variant-specific факты.

## Явно запрещённые вещи
- превращать guide в замену formal report;
- брать чужие примерные числа вместо чисел текущего варианта;
- пересказывать формулы без объяснения символов;
- перескакивать от математики к графику без словесного моста;
- делать defense help отдельным необязательным хвостом;
- подменять подтверждённые выводы свободной интерпретацией.
