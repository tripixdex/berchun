# Аудит входных материалов

## Назначение
Этот документ фиксирует, какие входы реально доступны в репозитории на Stage 01, какую роль они играют и что из них можно безопасно использовать в следующих стадиях.

## Примечание о путях
В пользовательской постановке были указаны пути `/mnt/data/...`, но в текущем окружении каталог `/mnt/data` отсутствует. Для аудита использованы локальные копии материалов из корня репозитория и локальные шаблоны-аналоги отчётов из другого учебного проекта.

## Извлечённые наблюдения по полезным источникам

### `my_var.md`
- Домашнее задание состоит из двух задач: `Проектирование Call-центра` и `Проектирование производственного участка`.
- Задача 1 разбита на четыре подпункта:
  - без очереди;
  - с ограниченной очередью;
  - с неограниченной очередью;
  - с неограниченной очередью и уходом клиентов.
- Для подпунктов 1.2, 1.3 и 1.4 явно указан верхний предел `до 15` по числу операторов; для подпункта 1.2 также явно указан верхний предел `до 15` по числу мест в очереди.
- Для подпункта 1.1 верхняя граница по числу операторов не фиксирована; требуется довести перебор до достижения вероятности отказа менее `1%`.
- Параметры варианта заданы только формулами:
  - для задачи 1: `Tc = 10 + номер по журналу`, `Ts = 40 + число дня рождения`, `Tw = 100 + номер месяца рождения`;
  - для задачи 2: `N = 30 + номер месяца рождения`, `Tc = 100 + номер по журналу`, `Ts = 25 + число дня рождения`.
- Числовые значения параметров варианта в репозитории отсутствуют.

### `ot_prepoda.md`
- Преподаватель дал ссылку на GPSS World Student Setup и рекомендовал заранее скачать текст референсной модели.
- Это контекст по инструментам курса, а не формальное дополнение к постановке варианта.
- Так как Stage 01 ограничен локальными входами, внешние материалы не скачивались и не использовались как источник фактов.

### `references/DZ1.docx`
- Это полноценный образец отчёта по тому же ДЗ.
- По текстовой экстракции обнаружены:
  - титульный лист;
  - раздел `Задача №1` с подпунктами `1.1`, `1.2`, `1.3`, `1.4`;
  - раздел `Задача №2`;
  - не менее `32` подписей к рисункам;
  - не менее `263` формульных объектов OMML;
  - не менее `35` встроенных медиа-изображений.
- По извлечённому тексту не обнаружены обязательные разделы `Таблица`, `Заключение`, `Список литературы`, `Приложение`.
- Образец показывает ожидаемую структуру каждого содержательного подпункта:
  - краткая постановка;
  - исходные данные;
  - расчётная схема;
  - формулы вероятностей состояний и производных метрик;
  - графики;
  - краткие выводы по характерным режимам.
- Подпункт `1.2` в образце оформлен как `12` семейств графиков плюс расчётная схема.
- Подпункт `1.3` явно содержит комментарий о потере стационарности при перегрузке системы.
- Подпункт `1.4` явно вводит интенсивность ухода из очереди и аппроксимацию/обрезание ряда для вычисления.
- Раздел `2` оформлен как расчётная схема плюс `5` графиков по числу наладчиков.
- Числа `Tc = 11`, `Ts = 68`, `Tw = 105`, `N = 35`, `Tc = 101`, `Ts = 53` относятся к чужому примеру и не могут быть перенесены в наш вариант.

### `references/DZ1.pdf`
- Содержательно дублирует образец отчёта.
- В текущем окружении отсутствует `pdftotext`, поэтому PDF использовался как подтверждение наличия второй копии образца, но не как основной источник текстовой экстракции.

### `references/Статья.docx`
- Это перевод обзорной статьи по внутренним механизмам ПО для дискретно-событийного моделирования.
- Статья полезна для понимания сущностей, ресурсов, календарей событий, логики ожидания и верификации модели.
- В статье нет формул, специфичных для нашего варианта; она является теоретическим фоном, а не прямым шаблоном решения.

### `references/DZ2/...`
- C++-проект моделирует производственный поток с типами деталей, маршрутами по станциям `A1/A2/A3`, календарями `CEC/FEC`, нормальными распределениями и логированием.
- Это не прямое решение ни задачи 1, ни задачи 2 из `my_var.md`.
- Модель полезна только как общий пример событийной симуляции и дисциплины логирования.
- Прямое переиспользование кода опасно:
  - предметная область не совпадает;
  - выходные метрики не совпадают;
  - проект захламлён бинарными и IDE-артефактами.

## Переиспользуемые артефакты
- Структура итогового учебного отчёта из `references/DZ1.docx`.
- Набор ожидаемых рисунков и их тематик из `references/DZ1.docx`.
- Титульная страница как ориентир по академическому оформлению из `references/DZ1.docx`.
- Формульная дисциплина и необходимость пояснять модель состояния из `references/DZ1.docx`.
- Стиль stage-отчётности из локальных шаблонов `MASTER_EXECUTION_REPORT.md` и `STAGE_07F64_REPORT.md`.
- Идея разделения календарей событий и логов из `references/DZ2/...` допустима только как архитектурный мотив, если позже вообще будет выбрана симуляционная ветка.

## Опасные предположения, которых нужно избегать
- Не подставлять в расчёты чужие значения из образца `DZ1.docx`.
- Не считать, что `references/DZ2` уже решает задачу `Проектирование производственного участка`.
- Не считать GPSS обязательным инструментом без явного подтверждения; в доступных источниках это рекомендованный курсом софт, а не зафиксированное требование к сдаче.
- Не считать текстовую экстракцию DOCX полной заменой точной транскрипции формул; для финального отчёта при необходимости формулы нужно брать из исходного DOCX/XML, а не восстанавливать на глаз.
- Не придумывать отсутствующие исходные параметры варианта: номер по журналу, день рождения, месяц рождения.
- Не смешивать подпункт `1.1` с подпунктом `1.2`: случай нулевой очереди уже выделен в отдельную задачу.
- Не вводить симуляцию как самоцель, если аналитический расчёт покрывает требования точнее и проще.
